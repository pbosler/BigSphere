11 This software provides a Lagrangian Particle-Panel Method for solving the barotropic vorticity equation
12 and the advection equation on the sphere and in the plane.
13 
14 A shallow water equation solver is in development.

16 Design / Use
17 =========
18 
19 Although it is written in modern Fortran rather than C++, the code is organized in an object-oriented style.
20 This means that for each task the code performs, there is a derived data type that corresponds to that task (like a C++ class).
21 Each module file contains one derived data type and its associated methods.
22 For example, PlaneMesh.f90 contains the data type for moving meshes in the plane and interfaces to all the functions that are
23 implemented on PlaneMesh objects.
24 The PlaneDirectSum.f90 module defines the Runge-Kutta time stepping data type for plane meshes, and contains all of the timestepping functions.
25 
26 Each object must be created in memory using its New method, many objects must then be initialized using a call to an Init or Initialize subroutine.
27 For clean code, every time a New subroutine is called on an object, a corresponding call to that object's Delete subroutine should be used when the object is no longer needed or goes out of scope.
28 
29 Example:
30 ---------
31 To define a Rossby-Haurwitz wave vorticity distribution for a sphere mesh, a user would first allocate the VorticitySetup object by calling its New subroutine.
32 Then the wave would be initialized with user-defined parameters (most commonly read from a namelist file) by calling the VorticitySetup module's InitRHWave routine.
33 Finally, the wave is defined and users can define the vorticity on a sphere mesh by calling the SetRHWaveOnMesh subroutine.
34 
35  call New(rhWave, nIntegers, nReals)
36  call InitRH4Wave(rhWave, backgoundWindSpeed, amplitude)
37  call SetRH4WaveOnMesh( sphere, rhWave)
38 
39 Code runs...
40 
41  call Delete(rhWave)

43 To create a new test case:
44 ---------------
45 The first step is to create the object that will control your test case, either a tracer or a vorticity distribution, in the Tracers.f90 or BVEVorticity.f90 files.
46 Test cases are defined by BVESetup or TracerSetup objects.
47 These objects hold the integers and real numbers that define your test case; numbers that, for example, define the amplitude of a Rossby-Haurwitz wave or the maximum value of an advected tracer.
48 To create the object the first step is to allocate the memory to hold these parameters using the "New" subroutine.
49 
50  call New(rhWave, RHWAVE_N_INT, RHWAVE_N_REAL)
51 
52 RHWAVE_N_INIT and RHWAVE_N_REAL are integer constants defined in the BVESetup.f90 module; their values are 0 and 2 for the RH4 test case,
53 which indicates that this test case is defined by 2 real numbers (the ampilitude and phase speeed of the wave).
54 Now that memory is allocated to hold these parameters, the test case must be initialized from user input (typically read from a namelist file).
55 If the variables backgroundWindSpeed and amplitude are set by the user, the test case is defind by calling the "Init" subroutine on the test case object.
56 
57  call InitRH4Wave(rhWave, backgroundWindSpeed, amplitude)
58 
59 Finally and most importantly is the subroutine which translates a test case into a tracer or vorticity distribution over a mesh.
60 These subroutine must all have the same interface -- only 2 arguments, the first being a mesh object and second being the test case definition.
61 This interface is required in order for the LagrangianRemeshing subroutines to work properly.
62 
63  call SetRH4WaveOnMesh( aMesh, rhWave)
64 
65 SetRH4WaveOnMesh calls the mathematical functions that define the vorticity distribution of a Rossby-Haurwitz wave, given the parameters defined by the rhWave test case object.
66 
67 To create a new test case then, users must first decide how many integers and reals are required to define their test case, and allocate the appropriate amount of memory with the "New" subroutine.
68 Then the test case object must be initialized with an "Init" subroutine to define the parameters in the test case object's memory.
69 Finally, users must write a SetTestCaseOnMesh subroutine and the associated mathematical functions that define their test case.

72 Requirements
73 =========
74 The code requires a modern Fortran compiler with an OpenMPI (<http://www.open-mpi.org>) distribution for parallel computing.
75 Without OpenMPI serial computations are still possible, but the code is designed to use distributed memory parallelism.
76 
77 Currently the software is written for Intel compilers and makes use of the Intel MKL software library (LAPACK, specifically).
78 It is possible to use other compilers (e.g. gfortran) provided OpenMPI and LAPACK are also available.
79 
80 LPPM uses interpolation software provided by ACM TOMS algorithm 526 written by Hiroshi Akima and implemented by John Burkhardt (<https://orion.math.iastate.edu/burkardt/f_src/bivar/bivar.html>),
81 and ACM TOMS algorithms 772 and 773 written by Robert Renka. All other code is original and authored by Peter Bosler.
